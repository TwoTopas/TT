# 3-Phase New Project Launch Framework（实践记录 2026-06-25）

## 场景

基于旧版全角色8维度评审输出的问题清单，从零启动全新微信小程序项目。新旧版本完全物理隔离。

## 执行流程

```
Phase 1: 旧版全量问题复盘 → 《新项目前置规避约束清单》
Phase 2: 新版 PRD V1.0 → 7大模块完整输出
Phase 3: 全角色开发落地全案 → Claude Code 分批执行
```

## Phase 1 产出物

文件：`phase1-problem-audit.md` (15.8KB)

内容结构：
- 9类问题分类提取（产品需求/交互体验/技术架构/后端实现/前端性能/质量测试/安全合规/运维交付/项目管理）
- 逐条根因分析（需求设计缺陷/技术选型失误/流程机制缺失/平台规则忽略/资源投入不足）
- P0/P1/P2 重定级
- 《新项目前置规避约束清单》表格（25条P0红线）
- 旧版P0问题闭环确认表

## 执行方式

Hermes 写 spec → delegate_task 创建页面（2个subagent并行）→ Hermes 全量语法验证

| 批次 | 内容 | 工具 |
|:-----|:-----|:-----|
| Batch 1 | 项目骨架+工具层(6文件) | write_file（Claude Code pipe超时后退回） |
| Batch 2 | 4个组件(16文件) | write_file（组件逻辑简单直接写） |
| Batch 3 | 7页面+2云函数(24文件) | delegate_task并行委派（2 subagent） |
| Batch 4 | 全量语法验证 | `node -c` 18/18通过 |

## 关键教训

1. **Claude Code pipe 首次尝试即超时** → 立即回退 write_file，不重试
2. **delegate_task 的 context 必须包含完整设计规范值**（色值、字号、间距），不能只写"参考XX skill"
3. **subagent 完成后必须立即做 `node -c` 全量语法验证**，subagent自检报告不可信
4. **18个JS文件全部通过语法检查**才算是验证通过
5. **全项目最终 4,120行/48文件，耗时约 2 小时完成**
6. **微信 require 不支持 `/` 绝对路径** — 必须用相对路径 `../../../`
