# 深度讨论 + 批量优化执行管线（2026-06-24 session验证）

> 从多角色深度讨论到批量代码执行的全流程模版。

## 完整流水线

```
测试工程师出报告 → orchestrator组织深度讨论
    → 专家基于测试结果展开（非独立输出）
    → 主持人裁决冲突
    → 输出 final-prompt.txt（可执行指令）
    → Hermes 按批次执行优化（patch/write_file）
    → 全量语法验证 + WXML检查
```

## Orchestrator Prompt 模版（已验证可行）

```python
delegate_task(
    role='orchestrator',
    goal='组织XXX优化的多角色深度讨论。流程：1. 先读测试结果 2. 测试工程师发言 3. 产品经理排优先级 4. 架构师评估可行性 5. 设计师出方案 6. 主持人裁决，输出final-prompt.txt',
    context=f'''## 讨论背景
[当前状态/测试结果/已发现问题]
[精确到文件行号]

## 讨论流程
1. **测试工程师**基于测试结果，列出最优先要修的Bug（精确到文件行号）
2. **产品经理**从付费转化角度排优先级
3. **前端架构师**评估每个方案的技术可行性
4. **UI/UX设计师**给具体设计方案（精确到色值rpx）
5. **主持人裁决**冲突，输出可执行prompt

## 相关SKILL
[在此列出相关skill名称和关键规范]

## 关键规则
- 各专家必须基于前一个人的输出展开
- 最终产出必须是给Claude Code的可执行prompt（不是分析报告）
- 有冲突时必须裁决给出最终方案
- 输出写入 final-prompt.txt''',
    toolsets=['file', 'terminal']
)
```

## 输出格式（final-prompt.txt）

最终输出必须是精确到文件路径+行号+修改内容的可执行指令：

```markdown
# 优化任务书

## 步骤1：XXX
**文件：** pages/xxx/xxx.wxml 第N行
**当前：** `旧代码`
**改为：** `新代码`

## 步骤2：XXX
...
```

## 关键规则（session教训）

1. **测试驱动讨论** — 必须先生成测试报告（5项全覆盖：JS语法、模块导出、WXML合规、数据完整性、付费墙逻辑），专家基于测试结果展开，不能空谈
2. **冲突必须有裁决** — 设计师想用方案A，架构师说做不了，主持人必须当场给方案B（不是搁置）
3. **精确到色值rpx** — 所有设计建议必须给具体值，不能只说"统一配色"
4. **batch执行** — 最终prompt列出10步左右，按执行顺序排列，每步精确到文件路径+行号
5. **执行方式** — 简单颜色替换用execute_code批量改，JS/WXML重写用write_file/patch，不经过Claude Code（超时率>80%）
