# 8角色全栈产品评审框架

> 当用户要求对产品进行全维度深度审查时，按顺序执行8个专业角色的独立分析。
> 每个角色从各自专业视角输出: 核心辩驳结论 → 深度辩驳与论证（带问题等级） → 分级优化方案（P0/P1/P2）。

## 调用时机

用户说"请代入XX年XX师身份"或"从XX视角评估产品"时。不预设顺序——按用户指定的角色执行即可。
当用户说"所有角色都来一遍"时，按以下顺序串行执行（不可并行，因为每个角色的输出可能引用前一个角色的发现）：

## 8角色执行顺序

```
1. 产品经理 (Product Manager)     — 产品定位、用户需求、优先级、竞品
2. 业务分析师 (Business Analyst)   — 业务流程闭环、规则严谨性、效率
3. UX设计师 (UX Designer)          — 交互逻辑、视觉层级、一致性、无障碍
4. 系统架构师 (System Architect)   — 架构选型、模块划分、高可用、扩展性
5. 后端工程师 (Backend Developer)  — 接口设计、数据处理、日志、安全
6. 前端工程师 (Frontend Developer) — 渲染性能、组件复用、兼容性、异常边界
7. 测试工程师 (QA Engineer)        — 边界测试、并发测试、可测试性、回归风险
8. SRE/运维工程师 (SRE)            — 部署流程、监控告警、容量规划、容灾
```

## 每个角色的输出结构（必须遵循）

```
### 核心辩驳结论
[一段直击要害的总论，打分或定性评价]

### 深度辩驳与论证
1. 【P0/P1/P2等级】【问题标题】
   - 问题定性+具体论证
   - 影响范围与严重程度量化
   - （可选）代码/数据引用
2. 同上...

### 分级优化方案
- P0 必改项: [具体方案，含修改路径、预期收益、成本]
- P1 建议项: ...
- P2 迭代项: ...
```

## 问题等级定义

| 等级 | 含义 | 响应时间 | 影响范围 |
|:----:|:-----|:--------:|:---------|
| P0 | 阻塞业务运转/致命安全漏洞 | 立即 | 全局 |
| P1 | 严重影响体验/效率/稳定性 | 本周 | 模块/功能 |
| P2 | 影响可维护性/可扩展性 | 本月 | 技术债 |

## 评审原则

1. **先体验再评估** — 不是凭空分析，而是先读取真实代码/页面
2. **具体到代码行/文件路径** — 所有批评必须有引用，不能空泛说"不好"
3. **量化影响** — "影响转化率15-20%" 而非 "影响体验"
4. **方案要可行** — 给出具体修改方案和预估工时
5. **辨驳在前，方案在后** — 先证明这是个问题，再给出怎么修
