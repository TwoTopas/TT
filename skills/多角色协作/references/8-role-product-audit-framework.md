# 8角色产品全量审计框架

> 2026-06-25 session：对「开店助手」小程序执行了8角色全量评审
> 用途：新项目启动前对旧版做系统性问题盘点，确保新项目从根源规避

## 适用场景

- 需要对新项目做建项前的全量问题盘点
- 旧版存在多个已知问题，需要系统性地在新版中前置规避
- 团队需要一份跨角色的统一问题基线

## 8角色评审流程

### 阶段1：独立评审（每个角色独立输出）

| 角色 | 评审视角 | 核心问题 |
|:-----|:---------|:---------|
| 产品经理(10年) | 产品定位、价值主张、转化漏斗 | 定位偏差、定价策略、付费墙断层 |
| 业务分析师(8年) | 业务流程闭环、规则严谨性 | 支付链路、数据持久化、合规风险 |
| UX设计师(10年) | 交互逻辑、视觉层级、一致性 | 设计系统混乱、信息层级颠倒、组件复用 |
| 系统架构师(10年) | 模块划分、数据流转、扩展性 | 3套支付架构并行、云函数未部署、废弃代码 |
| 后端开发(8年) | 接口设计、参数校验、日志 | 无校验层、markPaid无幂等、零日志 |
| 前端开发(8年) | 渲染性能、状态管理、异常边界 | 付费内容硬编码、NaN传播、无防抖 |
| 测试工程师(8年) | 边界场景、可测试性、回归风险 | 零测试、rent=0满分、并发双击无防护 |
| SRE(8年) | 部署、监控、容量、容灾 | 无部署流水线、零监控、无环境隔离 |

### 阶段2：问题统一归类

按以下9类对问题归档，统一重定级：

| 类别 | 标签 | 典型问题 |
|:-----|:-----|:---------|
| 产品需求类 | PR- | 定位偏差、功能缺失 |
| 交互体验类 | UX- | 设计系统、信息层级 |
| 技术架构类 | TA- | 架构分裂、数据流转 |
| 后端实现类 | BE- | 接口设计、校验缺失 |
| 前端性能类 | FE- | 性能瓶颈、状态管理 |
| 质量测试类 | QT- | 测试覆盖、边界场景 |
| 安全合规类 | SC- | 付费绕过、数据泄露 |
| 运维交付类 | OP- | 部署、监控、容灾 |
| 项目管理类 | PM- | 范围管控、迭代节奏 |

### 阶段3：重定级与规避约束

| 级别 | 定义 | 新项目要求 |
|:----:|:-----|:-----------|
| P0 | 阻塞业务闭环，涉及安全/合规/付费 | 必须100%在设计阶段前置规避 |
| P1 | 严重影响用户体验或运营效率 | 必须在开发阶段前置规避 |
| P2 | 影响迭代效率或长期可维护性 | 纳入后续迭代规划 |

### 阶段4：《新项目前置规避约束清单》

每条P0问题输出一条约束，格式：

| 旧版编号 | 根因 | 规避阶段 | 强制执行要求 |
|:---------|:-----|:---------|:-------------|
| PR-01 | 需求设计缺陷 | 需求阶段 | 新项目价值主张必须为正面叙事 |

### 阶段5：闭环确认表

终版输出《旧版P0问题闭环确认表》：

| 旧版P0问题 | 新版落地文件 | 验收状态 |
|:-----------|:------------|:--------:|
| 付费内容硬编码 | 云函数下发，搜索0处前端硬编码 | □ |

## 典型输出结构

```
# Phase 1: 旧版全量复盘 → 规避约束清单
## 1.1 全部问题提取与分类（按9类归档）
## 1.2 新项目前置规避约束清单（P0每条有对应约束）
## 1.3 旧版P0问题闭环确认表

# Phase 2: 新版PRD V1.0
## 2.1 项目概述（价值主张+历史规避总览）
## 2.2 平台生态规划
## 2.3 功能需求清单（标规避旧版XX问题）
## 2.4 核心业务流程（含异常分支）
## 2.5 页面与交互说明
## 2.6 非功能需求
## 2.7 验收标准

# Phase 3: 全角色开发落地全案
## 9角色交付物（PM/UX/架构/后端/前端/测试/安全/SRE/收尾）
## Claude Code执行指令汇总（分批次）
```

## 关键坑

1. **不要各自独立输出** — 产品经理写A、设计师写B、两人互不引用 = 表面评审。必须产品经理先输出 → 设计师基于其输出做体验评估 → 架构师基于前两者做技术评估。
2. **新项目完全独立** — 不继承旧版任何代码。旧版只是"问题复盘参考样本"。
3. **每项功能标注「对应规避旧版XX问题」** — 确保可追溯，不遗漏。
