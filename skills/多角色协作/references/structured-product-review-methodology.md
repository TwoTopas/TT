# 结构化产品评审方法论（5角色框架）

> 2026-06-25 session 验证的评审模式。
> 适用场景：当用户要求「从XX角色视角评价产品」时使用。

## 评审角色与结构

每次评审遵循统一的3段式结构：

**核心辩驳结论** — 一句话定性，指出最核心的问题。

**深度辩驳与论证** — 按编号列出问题，每个问题包含：
1. 问题定性（严重等级）
2. 专业论证（违背了什么原则/规则）
3. 量化影响（转化率/流失率/合规风险）
4. 影响范围

**P0/P1/P2分级优化方案** — 每项包含优化路径、预期收益、落地成本。

## 5个评审角色的侧重维度

| 角色 | 定位 | 核心关注 | 产出形式 |
|:----|:-----|:---------|:---------|
| 产品经理（10年） | 商业化与价值主张 | 定位偏差、转化漏斗、定价策略、用户路径 | 功能优先级、ARPU提升方案 |
| 业务分析师（8年） | 流程合规与闭环 | 业务规则漏洞、异常场景、合规风险、数据长尾 | 流程修复、审计清单 |
| UX设计师（10年） | 体验一致性与认知成本 | 交互心智、视觉层级、信息系统性、无障碍、异常态 | 设计系统、交互优化 |
| 系统架构师（10年） | 长期演进与可靠性 | 架构分裂、技术债、容灾、降级、可扩展性 | 收敛方案、技术路线图 |

## 评审清单

### 产品经理评审清单
- [ ] 价值主张是否明确且增益型？（而非规避型）
- [ ] 用户路径是否有超过3步才触及核心行动？
- [ ] 免费/付费内容的区分是否让用户感知到价值差？
- [ ] 定价是否有锚点/诱饵/套餐？
- [ ] 是否有「用完即走」还是「反复回来」的设计？

### 业务分析师评审清单
- [ ] 支付链路是否闭环？（支付→订单→标记→持久化→退款）
- [ ] 数据持久化是否跨设备/清缓存可恢复？
- [ ] 输入验证是否拦截空值/NaN/负数？
- [ ] 定价是否有虚假锚点合规风险？
- [ ] 审计日志是否存在？

### UX设计师评审清单
- [ ] 是否有统一的设计系统（背景/主色/圆角/间距）？
- [ ] 同一组件在不同页面是否结构一致？
- [ ] 表单是否有实时验证和即时反馈？
- [ ] 付费弹窗是否有信任信号（退款保障/支付方式/社会证明）？
- [ ] 异常态覆盖：loading/empty/error/network？

### 系统架构师评审清单
- [ ] 是否存在多套并行方案互相冲突？
- [ ] 云函数是否已部署？环境ID是否配置？
- [ ] require路径是否都正确？（尤其是深度嵌套页面）
- [ ] 数据流是否耦合过紧？（globalData命名空间隔离？）
- [ ] 废弃代码是否已隔离或标记？

## 输出示例（P0/P1/P2分级）

| 等级 | 定义 | 响应期限 |
|:----:|------|:--------:|
| P0 | 阻断性——业务无法运转或存在合规/安全风险 | 立即 |
| P1 | 严重——效率/体验/可维护性受损，但不阻断核心流程 | 本周 |
| P2 | 优化——长期竞争力提升 | 可规划 |
