## 1. code-review-and-quality — 5轴审查框架

### 核心吸收点

#### 审查发现分级体系

| 级别 | 含义 | 作者必须做 |
|------|------|-----------|
| *(无前缀)* | Required change | 合入前必须处理 |
| **Critical:** | 阻塞合入 | 安全漏洞、数据丢失、功能损坏 |
| **Nit:** | 可选 | 格式/风格偏好，可以忽略 |
| **Optional:** / **Consider:** | 建议 | 值得考虑但不是必须 |
| **FYI** | 仅供参考 | 无需操作 |

#### 5轴审查维度

1. **Correctness（正确性）**
   - 是否匹配 spec/任务需求？
   - 边界情况处理了吗（null、空、边界值）？
   - 错误路径处理了吗（不只是 happy path）？
   - 测试通过了？测试是否测了正确的东西？
   - 是否存在 off-by-one、竞态条件、状态不一致？

2. **Readability & Simplicity（可读性）**
   - 命名是否清晰且符合项目惯例？（没有 `temp`、`data`、`result`）
   - 控制流是否直接？（避免嵌套三元、深层回调）
   - 代码组织是否符合逻辑？（相关代码分组、清晰模块边界）
   - **能用更少行数完成吗？**（1000行当100行足够 = 失败）
   - **抽象值得它带来的复杂度吗？**（不要等到第三个用例才通用化）
   - **新条件分支是否附着在无关流程上？** → 应独立为 helper 或模型
   - **同一模式重复的条件判断？** → 缺少模型或 dispatcher

3. **Architecture（架构）**
   - 是否遵循现有模式？如果是新的，有理由吗？
   - 是否保持清晰的模块边界？
   - 是否有可共享的代码重复？
   - 依赖方向是否正确（无循环依赖）？
   - **这次重构是减少了复杂度还是移动了复杂度？**
   - **功能逻辑是否泄漏到通用模块？**
   - **类型边界是否明确？**

4. **Security（安全）**
   - 用户输入验证和清理？
   - 密钥远离代码/日志/版本控制？
   - 鉴别/授权检查到位？
   - SQL 参数化？（不是字符串拼接）
   - 输出编码防止 XSS？
   - 外部数据源视为不可信？

5. **Performance（性能）**
   - N+1 查询模式？
   - 无边界循环或无限制数据获取？
   - 应该有 async 的同步操作？
   - UI 不必要的重渲染？
   - 列表端点缺少分页？

#### Structural Remedies（结构修复建议）

当发现问题时，**提出具体的重构名称**，不只说"太复杂"：

- **用类型模型或显式 dispatcher 替代条件链**
- **合并重复分支为单一清晰流程**
- **分离编排逻辑与业务逻辑**
- **将功能逻辑移出共享模块到所属包**
- **复用规范的 helper 而非自制的近重复**
- **让类型边界显式化，使下游分支消失**
- **删除不增加价值的透传封装**
- **拆分大文件为聚焦模块**

优先选择**移除活动部件**的修复方案，而非重新分布同样复杂度的方案。

#### Change Sizing

| 改动量 | 评价 |
|--------|------|
| ~100 行 | 好。一次审查能看完 |
| ~300 行 | 可接受，如果是一个逻辑变更 |
| ~1000 行 | 太大，需要拆分 |

**注意**：不仅看 diff 大小，也要看文件大小。一个小的 diff 可能把文件推到 1000 行以上，那时应先拆解再增补。

#### Dead Code Hygiene

重构后检查死代码。列出后**问用户再删**（不要沉默删除）：

```
DEAD CODE IDENTIFIED:
- formatLegacyDate() in src/utils/date.ts — 已被 formatDate() 替代
- LEGACY_API_URL in src/config.ts — 无处引用
→ 安全删除这些？
```

#### 审查速度

- 1 个工作日内响应
- 优先快速单次反馈而非一次就批准
- 大 PR 要求拆分，不要硬看

#### Honesty in Review

- 不要 rubber-stamp（"LGTM" 缺乏审查证据）
- 不要软化问题（"maybe a minor concern" 当实际是生产级 bug）
- 量化问题（"这个 N+1 查询每项加 50ms" 好于 "可能慢"）
- 接受覆写：如果作者有完整上下文并不同意，尊重他们的判断

## 2. doubt-driven-development — 质疑驱动开发

### 核心思想

> 自信的答案不等于正确的答案。长 session 会把假设静默变成"事实"。

**这不是 `/review`**（review 是对成品下结论）。这是**飞行中**姿态：非平凡决策在矫正还便宜时被交叉审查。

### 什么时候是"非平凡决策"

任意一项为真：

- 引入或修改分支逻辑
- 跨模块或服务边界
- 断言类型系统/编译器无法验证的属性（线程安全、幂等、排序、不变量）
- 正确性依赖未来读者看不到的上下文
- 影响范围不可逆（生产部署、数据迁移、公开 API 变更）

**不需要用**：重命名/格式化/文件移动、明确的用户指令、纯读代码、一行改动、纯工具操作。

### 流程：5步循环

```
[ ] Step 1: CLAIM — 写下主张 + 为什么重要
[ ] Step 2: EXTRACT — 隔离产物 + 契约，剥离推理过程
[ ] Step 3: DOUBT — 调起新上下文审查者，用对抗性 prompt
[ ] Step 4: RECONCILE — 对照产物文本逐条分类发现
[ ] Step 5: STOP — 满足停止条件（琐碎发现/3轮/用户覆写）
```

#### Step 1: CLAIM

```
CLAIM: "新的缓存层在读重负载下是线程安全的"
WHY THIS MATTERS: 竞态在此处破坏用户数据，QA 难以检测
```

写不出紧凑的 claim → 你只有感觉，不是决策。先显式化再审查。

#### Step 2: EXTRACT

给审查者**产物 + 契约**，不要给推理过程。

- 代码：diff 或函数——不是整个文件
- 决策：3-5 句加上必须满足的约束
- 断言的证据保持独立

剥离推理。如果你把结论也传过去，得到的将是结论的确认。

#### Step 3: DOUBT — 对抗性审查

审查者的 prompt **必须是 adversarial**。框架决定答案：

```
对抗性审查。找出这个产物的问题。
假设作者过度自信。寻找：
- 未声明的假设
- 未处理的边界情况
- 隐藏耦合或共享状态
- 契约可能被违反的方式
- 可能破坏的现有约定
- 意外输入下的失效模式

不要验证。不要总结。找出问题，或明确说出经过彻底检查找不到任何问题。

ARTIFACT: <粘贴产物>
CONTRACT: <粘贴契约>
```

**只传 ARTIFACT + CONTRACT。不要传 CLAIM。** 传结论会偏向认同。

#### Step 4: RECONCILE

审查者的输出是**数据**，不是判决。按优先级分类：

1. **契约误读** — 你提供的契约不清晰或不完整。先修契约
2. **有效且可操作** — 需要改动。改动后重循环
3. **有效但取舍** — 问题真实但修复成本 > 接受成本。显式记录取舍
4. **噪音** — 审查者在缺少上下文的条件下误判了。跳过

#### Step 5: STOP

- 下一轮只有琐碎或已考虑过的发现，**或**
- 3 轮完成（上报给用户，不要自己跑第 4 轮），**或**
- 用户明确说"发货"

#### 跨模型升级

单模型审查者会分享原作者的盲点。每个交互周期都应**显式询问用户**：
> "单模型审查已完成。想要第二个跨模型意见吗？选项：Gemini CLI、Codex CLI、手动外部审查、或跳过。"

**非交互上下文**中：跳过跨模型，但**必须在输出中声明**："跨模型跳过：非交互上下文。"

## 3. security-and-hardening — 安全加固

### STRIDE 威胁模型

| 威胁 | 问 | 典型缓解措施 |
|------|-----|-------------|
| **S**poofing（欺骗） | 有人能冒充用户/服务吗？ | 认证、签名验证 |
| **T**ampering（篡改） | 数据能在传输或存储中被更改吗？ | 完整性检查、参数化查询、HTTPS |
| **R**epudiation（抵赖） | 操作能以后被否认吗？ | 安全事件审计日志 |
| **I**nformation disclosure（信息泄露） | 数据能泄露吗？ | 加密、字段白名单、通用错误提示 |
| **D**enial of service（拒绝服务） | 系统能被压垮吗？ | 限流、输入大小限制、超时 |
| **E**levation of privilege（权限提升） | 用户能获得不应有的权限吗？ | 授权检查、最小权限 |

### 三级边界系统

| 级别 | 内容 |
|------|------|
| **Always Do（无例外）** | 所有外部输入在边界验证、SQL 参数化、输出编码防 XSS、HTTPS、密码 bcrypt/scrypt/argon2、安全头（CSP/HSTS）、httpOnly/secure/sameSite cookie、npm audit |
| **Ask First（需要审批）** | 新认证流程、新敏感数据类别、新外部服务集成、改 CORS、文件上传处理、改限流、提升权限 |
| **Never Do** | 提交密钥到版本控制、日志记录敏感数据、信任客户端验证、禁用安全头、eval/innerHTML、localStorage 存认证 token、暴露堆栈跟踪 |

### AI/LLM 安全（OWASP LLM Top 10）

- 所有模型输出视为不可信输入（LLM05）
- 假设 prompt 可被劫持（LLM01）
- 密钥和其他用户数据远离 prompt（LLM02/LLM07）
- 限制工具和 agent 权限（LLM06）
- 约束消耗（token/请求/循环深度上限）（LLM10）
- 隔离 RAG 向量存储（LLM08）

## 4. performance-optimization — 性能优化

### 核心原则

> **先测量再优化。** 不做测量的优化是猜测。

### 5步优化流程

```
1. MEASURE → 用真实数据建立基线
2. IDENTIFY → 找到实际瓶颈（不是假设的）
3. FIX      → 处理特定瓶颈
4. VERIFY   → 再测量，确认改善
5. GUARD    → 添加监控或测试防止回归
```

### Core Web Vitals 阈值

| Metric | Good | Needs Improvement | Poor |
|--------|------|-------------------|------|
| LCP | ≤ 2.5s | ≤ 4.0s | > 4.0s |
| INP | ≤ 200ms | ≤ 500ms | > 500ms |
| CLS | ≤ 0.1 | ≤ 0.25 | > 0.25 |

### 从哪里开始测量（症状树）

```
加载慢？
├── 大 bundle？ → 量 bundle 大小，检查 code splitting
├── 慢服务器响应？ → 量 TTFB
│   ├── DNS 慢？ → dns-prefetch/preconnect
│   ├── TCP/TLS 慢？ → HTTP/2, keep-alive
│   └── Server 处理慢？ → 分析后端，检查查询和缓存
└── 渲染阻塞资源？ → 检查 CSS/JS 阻塞
```

### 常见的借口 vs 现实

| 借口 | 现实 |
|------|------|
| "以后优化" | 性能债是会复利的。现在修复明显的反模式 |
| "我机器上很快" | 你的机器不是用户的机器 |
| "这个优化很明显" | 没测量过？你不确定 |
| "100ms 用户不会注意到" | 研究表明 100ms 延迟影响转化率 |
| "框架处理性能" | 框架不能修 N+1 查询或过大的 bundle |
