# 决策自检与质疑驱动

> 源自 Addy Osmani agent-skills 的 doubt-driven-development，适配 Hermes 规划阶段。

## 核心理念

自信的答案不等于正确的答案。长 session 会把假设静默变成"事实"。质疑驱动是在矫正还便宜时（规划阶段），用新上下文审查者交叉检验非平凡决策。

**这不是 review。** review 对成品下结论。这是**飞行中姿态**：每次非平凡决策在定稿前被对抗性质疑。

## 什么时候用

### 必须用（非平凡决策）

以下任意一项为真：

- 引入或修改分支逻辑
- 跨模块或服务边界
- 断言类型系统/编译器无法验证的属性（线程安全、幂等、排序、不变量）
- 正确性依赖未来读者看不到的上下文
- 影响范围不可逆（生产部署、数据迁移、公开 API 变更）

### 不要用

- 纯机械操作（重命名、格式化、文件移动）
- 明确的用户指令
- 纯读代码或总结
- 一行改动
- 用户明确要求速度优先于验证

如果每步都质疑，你什么都发不出。只对非平凡决策应用。

## 5步自检流程

```
[ ] Step 1: CLAIM — 写下主张 + 为什么重要
[ ] Step 2: EXTRACT — 隔离产物 + 契约，剥离推理过程
[ ] Step 3: DOUBT — delegate_task 调对抗性审查者
[ ] Step 4: RECONCILE — 对照产物文本逐条分类发现
[ ] Step 5: STOP — 琐碎发现/3轮/用户覆写
```

### Step 1: CLAIM — 显式化决策

在一到三行内写下核心主张：

```
CLAIM: "用本地缓存替代每次API查询，收益大于引入副作用的成本"
WHY THIS MATTERS: 选错了再改成本高，且用户能感知性能下降
```

写不出紧凑的 claim → 你只有感觉，不是决策。先显式化再审查。

### Step 2: EXTRACT — 最小可审查单元

给审查者 **产物 + 契约**，不要给推理过程。

- 决策：3-5 句方案描述 + 必须满足的约束条件
- 代码：diff 或函数，不是整个文件

剥离推理。如果你把结论也传过去，得到的将是结论的确认。

### Step 3: DOUBT — 调起对抗性审查

用 `delegate_task` 调起一个隔离的子 agent，只传 ARTIFACT + CONTRACT。

对抗性 prompt 模板（复制到 context 中）：

```
对抗性审查。找出这个决策/方案的问题。
假设作者过度自信。寻找：
- 未声明的假设
- 未处理的边界情况
- 隐藏耦合或共享状态
- 约束可能被违反的方式
- 现有约定/模式可能被破坏的风险
- 意外输入下的失效模式

不要验证。不要总结。找出问题，或明确说出经过彻底检查找不到任何问题。

ARTIFACT: <粘贴产物>
CONTRACT: <粘贴约束条件>
```

**传给子 agent 时，只传 ARTIFACT + CONTRACT + 对抗性 prompt。不要传你的 CLAIM。** 传结论会偏向认同。

调用示例：

```
delegate_task(
  goal="对抗性审查下面的方案，找出潜在问题",
  context="""对抗性审查。...
  ARTIFACT: ...
  CONTRACT: ..."""
)
```

### Step 4: RECONCILE — 分类发现

审查者的输出是**数据**，不是判决。你是编排者，对照产物原文逐条分类：

按优先级：

1. **契约误读** — 你提供的约束不清晰或不完整。先修契约，下轮继续
2. **有效且可操作** — 需要改动方案。改动后回 Step 1
3. **有效但取舍** — 问题真实但修复成本 > 接受成本。显式记录取舍给用户看
4. **噪音** — 审查者缺少上下文误判了。跳过

### Step 5: STOP — 有界循环

- 下一轮只有琐碎/已考虑过的发现 → 通过
- 3 轮完成 → 上报用户："3轮后仍有实质性问题，方案可能不够成熟"
- 用户明确说"就这个方案" → 通过

**不超过 3 轮。** 如果第 4 轮还有实质性问题，说明方案本身不成熟，不是审查不够。

```
Hermes 规划阶段            CC 执行后
  本 skill               /code-review

```

## 常见陷阱

| 借口 | 现实 |
|------|------|
| "我很确定，跳过质疑步骤" | 在新问题上，自信和正确性相关性很差。确定感恰恰是盲点藏身之处 |
| "调子 agent 审查太贵了" | 生产中调试一个错误方案的代价更大。有界循环；bug 没有界 |
| "审查者只会吹毛求疵" | 只有范围不清时才会。约束 prompt 为"找出会使方案在约束下失败的问题" |
| "我最后用 review 统一审" | review 是事后门禁。质疑驱动在方向错误还便宜时纠正 |
| "审查者否定了，所以我错了" | 审查者缺少你的上下文——否定是信息，不是判决 |
| "两轮审查胜过一轮" | 第二轮要有新价值才做。琐碎发现时立即停止 |

## 红牌信号

- 对一行重命名也调质疑驱动
- 把审查者输出当判决，不对照产物原文重读
- 超过 3 轮不报用户
- prompt 写成"这个方案好吗？"而不是"找出问题"
- 时间压力下跳过质疑，但决策是高风险
- 质疑剧场：连续≥2轮审查者都有实质发现，但零发现被归类为可操作
- 定稿后才质疑 → 那叫 review，不是质疑驱动
- 传给审查者时漏了 CONTRACT
- 把 CLAIM 传给了审查者（偏向认同）
