# 小程序著作权/不正当竞争风险分析框架

## 适用场景

当产品与已有竞品在功能/命名/设计上相似时，按此框架逐层评估风险。

## 第一层：著作权（著作权法）

### 法律标准

| 元素 | 法律属性 | 能否被独占 |
|------|---------|:----------:|
| 功能概念 | 思想 | ❌ 不能 |
| 业务流程（步骤式向导） | 功能流程 | ❌ 不能 |
| 商业模式（免费/付费） | 方法 | ❌ 不能 |
| 具体配色方案 | 表达 | ✅ 能 |
| 具体UI组件样式 | 表达 | ✅ 能 |
| 文案措辞 | 表达 | ✅ 能 |

### 侵权认定三要件

1. **接触** — 被告是否接触过原告作品？（反编译APK → 构成接触）
2. **实质性相似** — 相似部分属于"表达"而非"思想"
3. **排除合理来源** — 相似是否因功能限制/行业标准导致

### 风险判定矩阵

| 对比项 | 高相似风险 | 低相似风险 | 判定标准 |
|--------|-----------|-----------|---------|
| 产品名称 | 参谋 vs 参谋 | 参谋 vs 助手 | 通用词（参谋/助手/专家）不受保护 |
| 配色 | #007aff + #f2f2f7 | #6C63FF + #f5f3ef | 色值不同 + 暖/冷色系不同 |
| UI组件 | 相同图表类型+布局 | 环图vs雷达图 | 组件类型相同但视觉表现不同 |
| 步骤流程 | 完全相同步骤数+名称 | 步骤数不同/名称不同 | 步骤式流程属功能思想 |  |
| 文案 | 逐句相同 | 语义相似但措辞不同 | 独立创作vs复制 |

### 关键防御点

- 产品定位为"互补层"而非"替代品"
- 核心差异点（如1377自有案例库）对方没有
- 数据源合法（高德API + 自有数据 + AI生成）
- 已做品牌名脱敏处理

## 第二层：反不正当竞争法

比著作权侵权更常被援引。核心条款：

| 条款 | 内容 | 风险场景 |
|:----:|------|---------|
| 第6条 | 混淆行为 | 名称/装潢/域名近似导致用户误认 |
| 第8条 | 虚假宣传 | 夸大功能/数据来源不实 |
| 第12条 | 互联网专项 | 恶意干扰他人产品/流量劫持 |

### 名称混淆评估

```text
竞品名：上上参谋
我方名：开店助手

对比维度：
1. 字数不同（4字 vs 4字）
2. 共享字"开"+"店" — 行业描述词，非显著性部分
3. 核心区分词"参谋" vs "助手" — 语义不同（出谋划策 vs 辅助工具）
4. 整体呼叫效果不同
→ 🟢 低混淆风险
```

### 降低混淆风险的实操措施

```
1. 改名：去掉与竞品共享的核心词
   ❌ 开店参谋 → ✅ 开店助手（参谋→助手，完全不同的词）

2. 颜色区分：视觉上彻底不同
   ❌ #007aff iOS蓝 → ✅ #6C63FF 紫蓝

3. 首页声明独立性
   「本工具与XX等第三方平台无关，数据来源于自有案例库及XX开放数据」

4. 差异化功能（竞品没有的）：
   - 1377案例库
   - 3个工具而非1个
   - AI动态生成报告
```

## 第三层：平台审核风险

小程序审核比法律诉讼更常发生。微信审核的敏感点：

| 审核项 | 注意事项 |
|--------|---------|
| 类目冲突 | 与已上线产品同类别时，可能因功能相似被拒 |
| 名称审核 | 不能与已有小程序名称近似，不能含禁用词 |
| 内容审核 | 付费内容需符合虚拟商品规范 |

### 应对策略

- 名称查重：提交前在 mp.weixin.qq.com 搜索确认无冲突
- 类目选择：选择最准确的类目，避免"综合"类
- 提前准备：功能截图 + 差异化说明 供审核参考

## 完整分析示例流程

```text
1. 收集事实
   - 反编译过对方APK = 接触成立
   - 早期代码注释写"仿上上参谋" = 不利证据
   - 当前配色/组件/数据源完全不同 = 有利证据

2. 逐层评估
   - 著作权：相似在功能层面（思想），非表达层面 → 🟢 低
   - 反不正当竞争：名称已改+声明已加+功能有差异 → 🟢 低
   - 平台审核：类目冲突待验证 → 🟡 中

3. 执行降低措施
   - ✅ 改名（参谋→助手）
   - ✅ 改配色（#007aff→#6C63FF）
   - ✅ 加独立性声明
   - ✅ 加差异化功能
   - ✅ 品牌名脱敏

4. 持续监控
   - 微信审核结果
   - 竞品方投诉
   - 用户混淆反馈
```
