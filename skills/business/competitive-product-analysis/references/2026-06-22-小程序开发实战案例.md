## 背景

在完成上上参谋竞品分析（2026-06-21）和HTML原型交付后，TT要求直接开发微信小程序。三个模块：选址评估（4步）、周边调研（6步）、成本核算（6步），40%免费+60%付费。

## 交付模式选择

### 网页方案（最终未选）

| 优势 | 劣势 |
|------|------|
| 现有HTML原型可直接上线 | 小程序分享卡片点击率远高于链接 |
| 零审核周期 | 微信内打开网页体验差（白屏/缓存） |
| 纯前端双击可用 | 微信支付需企业资质+备案域名 |

### 小程序方案（TT选择）

| 优势 | 劣势 |
|------|------|
| 原生分享卡片（点击率35-50% vs 链接20-30%） | 开发工具依赖Windows/Mac |
| 微信搜索免费获客 | 审核周期1-7天 |
| 小程序支付（成功率95%+） | 个人主体不能虚拟支付 |
| 下拉菜单留存 | WXML开发效率低于HTML |

### 多角色协作结论

> 产品经理：先网页验证（1-2周）再转小程序
> 前端架构师：混合方案（网页先行）
> 增长运营：直接小程序（获客效率高3-5倍）

最终TT决策：**直接小程序**。

## 技术坑记录

### 坑1：tabBar图标找不到 → 阻止启动

**现象：**
```
Error: app.json: ["tabBar"]["list"][0]["iconPath"]: "images/tab-assess.png" 未找到
```

**原因：** 开发者工具在编译时会检查tabBar引用的icon文件。如果文件不存在，编译失败，整个app无法加载。

**修复：**
1. 先在JS中用PIL生成48x48 PNG文件（`Image.new('RGBA', (48,48))`）
2. 保存到 `images/` 目录
3. 再写 `app.json` 引用它们
4. **或者：删除tabBar配置**，改用自定义底部导航组件

**顺序必须对：** 图片文件先存在 → 再写 app.json。如果反过来，开发者工具缓存了旧app.json，即使后来补了图片也可能认不到。

### 坑2：app.js 引用外部域名 → 编译报错

**现象：**
```
https://api.example.com 不在以下 request 合法域名列表中
```

**原因：** 子角色subagent创建的app.js中设置了`apiBaseUrl: 'https://api.example.com'`。

**修复：** 原型阶段的app.js必须零外部API调用。重写为最小版本：

```javascript
App({
  globalData: { /* all local data */ },
  onLaunch() { console.log('[App] started'); },
  setPaid(module) { /* localStorage-based paywall */ }
});
```

任何`wx.request`调用都必须在开发者工具后台配置合法域名，否则编译不通过。

### 坑3：WXML不支持 style 中的JS表达式

**这是最隐蔽的坑，编译错误不会指向具体行号。**

**现象：**
```
Error: 编译 .wxml 文件错误
```
不告诉你哪个文件哪一行。

**原因：** WXML编译器不支持 style 属性内的 any mustache 表达式，例如：

```html
<!-- 全部不合法 -->
<view style="border:2rpx solid {{cond?'#007aff':'#eee'}}"></view>
<view style="color:{{myColor}}"></view>
<view style="font-size:{{size}}rpx"></view>
```

**修复：** 在JS中预计算整个style字符串，bind为单个data属性：

```javascript
// JS中
const items = raw.map(i => ({
  ...i,
  cardStyle: i.selected 
    ? 'border:3rpx solid #007aff;background:#e8f0fe;'
    : 'border:2rpx solid #e5e5ea;'
}));
this.setData({ items });

// WXML中
<view style="{{item.cardStyle}}"></view>
```

**class 属性可以**使用mustache（这是唯一可以用的地方）：
```html
<view class="tag {{item.isActive?'tag-active':'tag-default'}}"></view>
```

### 坑4：WXML不支持方法调用

**现象：** 页面白屏/空白，无编译错误。

**原因：** 以下全部不合法：
```html
<text>{{item.name.toUpperCase()}}</text>           <!-- ❌ -->
<text>{{(a/b).toFixed(1)}}</text>                  <!-- ❌ -->
<text>{{'前缀'+item.name}}</text>                  <!-- ❌ 字符串拼接 -->
```

**修复：** 所有需要计算的值都在JS中预计算好：
```javascript
this.setData({
  report: {
    ...rawData,
    displayName: '前缀' + rawData.name,   // 字符串拼接在JS做
    formattedValue: (a/b).toFixed(1),      // 数学运算在JS做
  }
});
```

然后在WXML只用 `{{report.displayName}}` —— 简单路径访问。

### 坑5：开发者工具热重载不可靠

**现象：** 改了app.json（删除tabBar），但开发者工具仍然报tabBar错误。

**原因：** 开发者工具会缓存旧的项目配置。即使文件已修改，工具仍读取内存中的旧版本。

**修复：**
1. 关闭项目（文件→关闭项目）
2. 重新打开
3. 或者工具栏→清除→清除所有缓存→重新编译

### 坑6：基础库3.16.1灰度版超时

**现象：**
```
Error: SystemError (appServiceSDKScriptError) timeout
```

**原因：** 2026年6月基础库3.16.1处于灰度测试阶段，有已知超时bug。

**修复：** 工具栏→详情→本地设置→基础库版本→选择稳定版（如3.3.4或更低）。

### 坑7：Page JSON必须完整

每个页面需要4个文件：`.wxml` + `.wxss` + `.js` + `.json`。缺少任何一个，页面无法加载。`.json`即使简单也要有：
```json
{"usingComponents":{},"navigationBarTitleText":"页面标题"}
```

## Subagent生成代码的可靠性

| 文件类型 | 由谁创建 | 可靠性 |
|---------|---------|:------:|
| 架构文档（.md） | Subagent | ✅ 高 |
| 数据层（data.js/formulas.js） | Subagent | ✅ 高 |
| 云函数（cloudfunctions/*.js） | Subagent | ✅ 高 |
| 页面WXML/WXSS/JS | **必须父进程自己写** | ❌ 低 |
| 全局样式app.wxss | Subagent | ⚠️ 中（可能太大太复杂） |
| 页面JSON | 父进程或subagent | ✅ 高 |

**已验证结论（2026-06-22）：** subagent写页面代码一定会出错（错误域名、占位key、复杂JS逻辑导致页面空白、WXML语法错误）。父进程必须亲自动手写页面。
