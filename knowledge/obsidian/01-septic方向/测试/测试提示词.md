## PM（以Bill身份模拟）提示词

```
你叫 Bill，52岁，在德州开化粪池泵抽公司，1台车。
客户名单全在笔记本上，不会用电脑，手机只会打电话和看短信。
你儿子帮你注册了 SepticSaver，给了你一个链接。

现在以"第一次打开这个链接"的状态走完整流程。
在每个步骤停下来问："这个步骤让 Bill 困惑吗？"
"困惑"的定义：用户说出"这是什么？"或者"然后呢？"或者关掉页面。

流程（10步）：

1. 打开链接
   - 你看到的是什么页面？你知道这是哪里吗？你知道"code=xxx"是什么意思吗？
   - ⚡评估点：注册页 vs 直接进 Dashboard？

2. 登录/注册
   - 如果你儿子已经帮你注册过，你第二次打开还需要输密码吗？
   - ⚡评估点：记住登录功能是否工作？

3. 进入空状态
   - 你看到"Your Customers — No customers yet." —— 你的客户都在笔记本上，现在你该做什么？
   - 两个大按钮："Import" 和 "Add Customer"——你选哪个？
   - ⚡评估点：没有CSV文件时用户是否卡住？

4. 导入客户
   - 如果你有CSV文件：选文件 → 点 Start Import → 页面变成什么？
   - 导入完成后：你知道成功了吗？下一步该点哪里？页面有没有告诉你要做什么？
   - ⚡评估点：导入后是否自动跳转？有无toast反馈？

5. 手动添加
   - 点了 Add Customer，填了一个客户名，保存后回到哪里？
   - 加的那个客户还在吗？如果不在，你知道为什么吗？
   - ⚡评估点：添加→空状态→添加→死循环？

6. Dashboard（有数据时）
   - 你看到 "Due This Week"、"Served This Week"、"All Customers" 这些数字——你理解它们是什么意思吗？
   - "Estimated revenue this week"——这个数字怎么算的？可信吗？
   - 客户列表谁的顺序排最前面？你知道为什么那个客户要优先做吗？
   - ⚡评估点：统计卡片对店主是否有决策价值？

7. 到期客户弹窗
   - 进入页面后弹出一个窗，告诉你"N个客户到期了"——你看了第一反应是什么？
   - 弹窗里给你什么选项？你点了什么？弹窗关掉后还能找到那些到期客户吗？
   - ⚡评估点：弹窗频率（每7天一次）是否合适？内容是否驱动行动？

8. 加日历提醒
   - 蓝色提示"📅 Get reminded on your phone — tap to download .ics"
   - 你点了一下，手机下载了一个文件。然后呢？你知道现在该做什么吗？
   - 如果儿子不在身边，你能自己搞定吗？
   - ⚡评估点：.ics下载后的下一步引导是否缺失？

9. 试用期最后7天
   - 你打开页面，看到黄条 "⏰ Your trial ends in 7 days"
   - 你点 💎 Upgrade —— 看到了什么？$79和$147的区别你理解吗？
   - "Pay by PayPal: paypal.me/xxx" —— 你试试点这个链接，能打开吗？
   - ⚡评估点：付款流程是否通畅？$79是否让用户觉得值？

10. 重复访问
    - 过了一天，你再次打开同样的链接 —— 你还能直接进Dashboard吗？还是又跳转登录页？
    - ⚡评估点：localStorage记住登录是否工作？

输出格式：
  😤 困惑时刻 [步骤N]：[具体描述]
  严重性：P0（不修用户不走）/ P1（不修用户烦）/ P2（体验优化）
  💡 建议修复：[一句话方案]
```

## QA/测试工程师提示词

```
对整个系统做全量测试，包括所有API端点和前端UI。

=== 1. API 边界测试 ===
1.1 空字段
   - 空name → 返回400
   - 空address → 正常通过（降级为空字符串）
   - 空system_type → 降级为 pumping
   - 空last_service_date → 降级为今天
   - name缺失（完全不传）→ 返回422

1.2 超长字段
   - name 500字符 → 应正常通过（DB VARCHAR无限制）
   - phone 50字符 → 应正常通过

1.3 特殊字符
   - name = "Test' OR '1'='1"（SQL注入尝试）→ 参数化查询，不应崩溃
   - name = "<script>alert(1)</script>" → 前端escape
   - name = 空格 → 应被trim后拒绝（if not name.strip()）

1.4 重复操作
   - 重复标记同一客户（7天内）→ 不应报错
   - 重复撤销同一服务记录 → 不应报错
   - 重复添加同名同地址客户 → 更新而非新增

1.5 不存在ID
   - /service/999/undo → 返回400/422
   - /customer/999 → 返回404
   - 登录错误密码 → 返回错误

1.6 认证
   - 无 X-Access-Token header → 422
   - 错误 X-Access-Token header → 401
   - 读操作（GET /api/shop/{code}）→ 无需Token
   - UUID查询 → 无需Token
   - iCal下载 → 无需Token

=== 2. 数据一致性 ===
2.1 添加后Dashboard刷新
2.2 标记后Dashboard刷新
2.3 两个店铺数据完全独立
2.4 软删除后不在活跃列表，恢复后回到列表
2.5 CSV导入去重：重复行只更新不新增

=== 3. UI 元素可见性 ===
3.1 全局搜索 var(--apple-*) 残留 → 0个
3.2 每个页面在375px宽度下有底部安全区域 padding-bottom
3.3 所有按钮触控目标 ≥44px min-height
3.4 每个页面有loading/空/正常三种状态
3.5 Toast 3.5秒后自动消失
3.6 模态弹窗关闭后页面状态不受影响

=== 4. 用户路径覆盖 ===
4.1 注册→欢迎页→Import→Dashboard→超期弹窗→标记→详情→返回
4.2 注册→欢迎页→Skip→空状态→Add Customer→保存→Dashboard
4.3 关掉浏览器→再次打开→直接进Dashboard（不跳登录页）
4.4 无 code 参数 URL → 登录页 → 登录 → Dashboard
4.5 非法 code 参数 URL → 显示错误提示（不崩溃）
4.6 退出登录（Sign Out）→ 清除 localStorage → 回到登录页
4.7 同一浏览器打开第二个店铺链接 → 第二个店铺数据不污染第一个
4.8 手机号加了区号和不加区号（搜索/显示）

=== 5. 性能测试 ===
5.1 连续10次标记/撤销操作不崩溃
5.2 200个客户列表加载时间<2秒
5.3 10个店铺同时在线不互相影响
5.4 大照片上传（2MB）不导致内存泄漏

输出格式：
  ❌ P0 [问题] — [根因] — [建议修复]
  ❌ P1 [问题] — [根因] — [建议修复]
  ❌ P2 [问题] — [根因] — [建议修复]
  ✅ [通过测试项]
```

## UI/UX 设计师提示词

```
用 iPhone SE（375px宽度，iOS Safari）打开所有页面检查：

=== 1. 布局完整性 ===
```

## 后端测试提示词

```
对所有 API 端点做自动化测试（使用 python requests / curl）：

=== 1. 返回结构 ===
1.1 成功: {"ok": true, ...} ——所有写操作和读操作
1.2 失败: {"detail": "..."} ——所有认证错误、业务错误
1.3 所有字段类型一致（绝不出现 None 被序列化为 "None" 字符串）

=== 2. 认证 ===
2.1 写操作无 X-Access-Token → 422 (Field required)
2.2 写操作错误 X-Access-Token → 401 (invalid access token)
2.3 读操作（GET /api/shop/{code}, GET /api/ical/{code}）→ 无需Token
2.4 UUID自助查询（GET /api/customer/query/{uuid}）→ 无需Token
2.5 Webhook（POST /api/webhook/sms）→ 无需Token

=== 3. 业务逻辑 ===
3.1 系统类型周期（标记已服务后自动计算）：
     - grease → 3个月后
     - pumping → 60个月后（5年）
     - inspection → 12个月后
     - install → 12个月后
     验证：对每种类型添加客户→标记→检查next_due_date
3.2 50客户上限：
     - 免费版超过50个客户时，写API返回402
     - CSV导入超过50个时，被拒绝的行返回errors列表
3.3 24小时撤销窗口：
     - 服务记录创建24小时内 → 撤销成功
     - 超过24小时 → 撤销被拒绝
     - 重复撤销同一记录 → 不报错
3.4 去重（同名+同地址）：
     - 添加第二个同名同地址客户 → 更新(phone/system_type)而非新增
     - Dashboard客户数不变
3.5 同名店铺注册：
     - 同一名称注册两次 → 第二次被拒绝（含detail说明）

=== 4. 并发 ===
4.1 同店铺 + 同客户名并发标记 → 只有一个成功
4.2 同公司名并发注册 → 只有一个成功
4.3 同时5个API请求 → 全部正常响应（不崩不超时）

=== 5. 持久化 ===
5.1 重启后端后数据不丢失
5.2 SQLite WAL模式下的并发读不被阻塞
5.3 备份/恢复脚本正常工作

输出：
  ✅ 通过 | ❌ 失败
  每个失败项附带: 请求方法 + 路径 + 参数 + 期望结果 + 实际结果
```

## 市场角色测试提示词

```
以"化粪池店主"的视角审核产品的营销和定位：

=== 1. 定价测试 ===
1.1 $79/$147 vs $149/月（竞品Tank Track）
     - 你在竞品对比时，第一眼看出价差了吗？
     - 你会理解为"一次付清 = 竞品两周费用"吗？

1.2 创始会员"仅剩X个名额"
     - 这个计数器让你有紧迫感吗？还是觉得"假的"？
     - 如果计数器跳得很快（一周内从100降到30），你会信吗？

1.3 免费试用45天
     - 45天足够你决定吗？还是太长/太短？
     - 50客户限制在你看来是"够用"还是"刻意限制"？

=== 2. 获客路径测试 ===
2.1 假设你在Facebook化粪池从业者群组里看到一个人推荐SepticSaver
     - 你会点进去看吗？
     - 你希望看到什么信息（功能列表、定价、用户评价）？

2.2 假设你的同行给你推荐
     - 你信他推荐的信任度有几分？
     - 你会用他给的那个创始会员链接吗？

=== 3. 竞争定位测试 ===
3.1 一句话描述"SepticSaver是什么"——你看这句话能不能说动你？
3.2 对比一下：Tank Track $149/月 vs SepticSaver $79一次付清
     - 你觉得哪个"质量更好"？
     - 你会不会觉得"便宜没好货"？

输出：
  ✅ 营销信息清晰 | ❌ 营销信息模糊
  🟢 定价有竞争力 | 🟡 定价需调整 | 🔴 失竞争力
  💡 [建议]
```
